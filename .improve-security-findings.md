# Security Audit Findings — wabbajack-py

**Date:** 2026-03-26
**Scope:** FastAPI backend, auth, WebSocket, updater, installer, downloaders
**Auditor:** Claude (automated analysis pass)
**Status:** Analysis only — no code changes made

---

## Summary

The codebase has received a meaningful security hardening pass (session tokens, path validation, URL scheme checks, WebSocket input guards). The remaining issues are not catastrophic for a local-only tool, but two warrant prompt attention because they affect code execution or key confidentiality.

### 1. CRITICAL — Binary Self-Update with Optional Integrity Check

**File:** `src/wabbajack/updater.py`, lines 328–391 (`_update_binary`)

**Description:**
The binary update path downloads a new executable and atomically replaces the running binary. The SHA-256 checksum verification is *optional* — it only runs if a `.sha256` sidecar file exists at `{download_url}.sha256`. If that file is absent (e.g., a release was published without the sidecar, or the sidecar URL returns a non-200), the update proceeds unconditionally:

```python
sha_resp = requests.get(sha_url, timeout=10)
if sha_resp.status_code == 200:
    # ... verify ...
else:
    log.warning("No .sha256 sidecar file found, skipping integrity check")
    # proceeds to overwrite the binary anyway
```

The `download_url` itself comes from the GitHub Releases API response (`asset['browser_download_url']`). This URL is fetched over HTTPS, but the asset payload is downloaded and executed without any mandatory signature verification. If an attacker could tamper with a release asset (compromised GitHub account, supply chain attack, or MITM on the download CDN), they would achieve arbitrary code execution.

**Additional concern:** The `/api/update/apply` endpoint (api.py:436) is POST-protected by the session token middleware, which is correct. However, the update logic is also callable from the CLI (`cli.py:474`, `main update` command) with no additional confirmation prompt.

**Fix:**

1. Make the checksum check **mandatory** — abort if the `.sha256` file is absent or if fetching it fails:
   ```python
   # Fail loudly if sidecar is missing
   sha_resp = requests.get(sha_url, timeout=10)
   if sha_resp.status_code != 200:
       return {'success': False, 'message': 'No checksum file available; refusing update'}
   ```
2. Consider adding a GPG-signed manifest (e.g., `checksums.txt.asc`) and verifying it against a pinned public key bundled with the application.
3. For the dev-mode git pull path, at minimum verify the remote URL matches the expected `GITHUB_REPO` before executing `git pull`.

### 2. HIGH — Nexus API Key Stored and Transmitted in Cleartext via Multiple Paths

**Files:**
- `src/wabbajack/web/api.py`, lines 258–266 (`POST /api/auth/nexus/key`, `class NexusKeyRequest`)
- `src/wabbajack/web/api.py`, lines 174–211 (`POST /api/install/start`, `InstallRequest.nexus_key`)
- `src/wabbajack/web/auth.py`, lines 67–85 (`load_saved_token` — env var fallback)

**Description:**

**Path A — Install request body:**
`InstallRequest` has an optional `nexus_key: Optional[str]` field. If the frontend sends the key here, it appears in:
- FastAPI access logs (if debug logging is enabled with uvicorn)
- Any crash dump or `log.error(f"Install failed: {e}")` trace (line 204 of api.py)
- The `req` object serialized by Python's default `__repr__` if the object is ever printed

**Path B — Key submission endpoint:**
`POST /api/auth/nexus/key` accepts `{"key": "..."}` in the request body. This is correct (body, not URL parameter), but there is no validation of the key format before calling `save_token`. A key of 0 length or a very long string is accepted. The `set_nexus_token` function does validate by calling the Nexus API, which will reject a bad key — but the bad key is still written to keyring by `save_token` before `set_nexus_token` clears it (auth.py line 95 calls `set_nexus_token` after `keyring.set_password`, meaning a failed validation clears `_nexus_token` in memory but leaves the bad/empty key in keyring).

**Path C — Environment variable fallback:**
`load_saved_token` reads `NEXUS_API_KEY` from the environment (auth.py:83). This is fine for CLI use, but the web GUI calls `load_saved_token` on `GET /api/auth/nexus/status` (api.py:226). If `NEXUS_API_KEY` is set in the server environment, the key will be silently loaded and used for all subsequent installs — with no user-visible indication that the key came from an environment variable vs. a user action.

**Fix:**

1. Remove `nexus_key` from `InstallRequest` entirely. The API already falls back to `get_nexus_token()` (the in-memory module-level state) when the field is absent. There is no need for the frontend to transmit the key on every install.
2. Fix the keyring write-before-validate ordering in `save_token` / `set_nexus_token`: only write to keyring after validation succeeds.
3. In `load_saved_token`, log a visible INFO message when the key is loaded from the environment so users are not surprised.

### 3. HIGH — WebSocket Endpoint Has No Authentication

**File:** `src/wabbajack/web/ws.py`, line 77 (`@router.websocket("/ws")`)

**Description:**
The `/ws` WebSocket endpoint accepts connections from any origin with no token check. The HTTP auth middleware (web/__init__.py) only gates `POST`/`PUT`/`DELETE` on `/api/*`. WebSocket upgrades are `GET` requests and are excluded.

While the commands accepted over WebSocket are limited to a known allowlist (`cancel`, `pause`, `resume`, `skip_file`, `manual_complete`, `retry_failed`), they have real operational impact on an in-progress install. A malicious page loaded in the same browser (or a DNS rebinding attack where the attacker's page connects to `localhost:6969`) could:

- Send `cancel` to abort an install mid-way (leaving partially-placed files)
- Send `skip_file` / `manual_complete` with arbitrary filenames to manipulate the install state
- Subscribe to the install log stream, which leaks full filesystem paths, archive names, download URLs, and Nexus rate-limit counters

The log stream is the most sensitive — it reveals the user's directory layout and the Nexus username/premium status via log messages emitted in `set_nexus_token`.

**Fix:**

1. Require the session token on WebSocket connections, passed as a query parameter (the standard pattern since WebSocket `connect()` does not support custom headers):
   ```python
   @router.websocket("/ws")
   async def websocket_endpoint(ws: WebSocket, token: str = ""):
       if token != SESSION_TOKEN:
           await ws.close(code=4401)
           return
       await ws.accept()
       ...
   ```
   The frontend connects as `new WebSocket("ws://localhost:6969/ws?token=" + sessionToken)`.
2. Add an `Origin` check as a second layer: reject connections whose `Origin` header is set to anything other than the allowed localhost origins (same logic as `get_session`).

### 4. MEDIUM — Path Traversal Check Is Incomplete for `_place_file` (Symlink Race)

**File:** `src/wabbajack/installer.py`, lines 263–306 (`_place_file`)

**Description:**
The `_place_file` method has two layers of path traversal protection:

1. String-level check: rejects `..` in path components and absolute paths.
2. `dest.is_relative_to(self.output)` check.

The comment on the `is_relative_to` check says "no symlink resolution needed — the string check above already caught `..` and absolute paths." This is correct for the basic case, but it misses a TOCTOU (time-of-check/time-of-use) symlink race:

If a malicious modlist directive places a file named `mods/evil` (passes the string check), and between the `is_relative_to` check and the `shutil.copyfile` call an attacker creates `output_dir/mods/` as a symlink pointing to `/etc/` (or any writable system path), the copy will follow the symlink and write outside `output_dir`.

In practice this requires the attacker to control both the modlist content and have concurrent filesystem write access, so exploitability is low. However, Wabbajack modlists are third-party files downloaded from external sources; a maliciously crafted modlist could include carefully timed directives.

The more realistic variant: a modlist directive could set `To` to a path like `overwrite/../../../.bashrc` — but this is already caught by the `..` check. The remaining gap is symlinks created *after* the check.

**Fix:**

1. Use `Path.resolve()` to canonicalize `dest` *after* the mkdir, then re-check `is_relative_to`:
   ```python
   dest.parent.mkdir(parents=True, exist_ok=True)
   # Re-check after mkdir to catch any symlinks just created
   try:
       if not dest.resolve().is_relative_to(self.output.resolve()):
           log.warning(f"  Symlink escape blocked: {to_field}")
           return False
   except (OSError, ValueError):
       return False
   shutil.copyfile(src, dest)
   ```
2. This is a defense-in-depth measure; the existing string check already eliminates the most common attacks.

### 5. MEDIUM — `GET /api/session` Origin/Referer Gating Is Bypassable via Missing Header

**File:** `src/wabbajack/web/__init__.py`, lines 41–52 (`get_session`)

**Description:**
The `/api/session` endpoint (which vends the `SESSION_TOKEN`) gates access using Origin and Referer headers:

```python
if origin and origin not in allowed:
    return JSONResponse(status_code=403, ...)
if referer and not any(referer.startswith(a) for a in allowed):
    return JSONResponse(status_code=403, ...)
return {"token": SESSION_TOKEN}
```

The guards only fire `if origin` / `if referer` — i.e., they are skipped entirely when both headers are absent. This is intentional (the comment says "Allow: no origin (same-origin fetch)"), but it creates a gap:

- A DNS rebinding attack that convinces the browser to treat the attacker's domain as `127.0.0.1` will still send an `Origin: http://evil.example.com` header, which *is* caught.
- However, a server-side request (e.g., `curl http://localhost:6969/api/session`) or a fetch from a non-browser context sends no Origin or Referer at all and receives the token unconditionally.
- If the server is accidentally exposed on a non-loopback interface (e.g., user runs `wabbajack-py serve --host 0.0.0.0`), any host on the LAN or internet can fetch the session token and then issue arbitrary mutating API calls.

The CLI default (`--host 127.0.0.1`) is safe, but the option exists and is easily overridden. There is no warning in the code or documentation about the security implications of `--host 0.0.0.0`.

**Fix:**

1. Add a warning log when binding to a non-loopback address:
   ```python
   # cli.py serve command
   if host not in ('127.0.0.1', '::1', 'localhost'):
       log.warning(f"WARNING: Binding to {host} exposes the web UI to the network. "
                   f"The session token provides no protection against unauthenticated "
                   f"access from other hosts.")
   ```
2. Consider adding a `REMOTE_ADDR` / client host check in `get_session`: only vend the token to loopback clients regardless of the bind address.
3. Optionally, make the "no Origin header" path only succeed for loopback clients:
   ```python
   client_host = request.client.host if request.client else ""
   if not origin and client_host not in ("127.0.0.1", "::1"):
       return JSONResponse(status_code=403, content={"detail": "Forbidden"})
   ```

## Minor / Informational

These are not ranked in the top 5 but are worth noting:

**M1 — `GITHUB_REPO` is hardcoded to `pjschulz3004/wabbajack-py`** (updater.py:9). If the repo is ever transferred or forked, users running old binaries will fetch updates from the new owner's releases. Not a current vulnerability, but worth noting for long-term maintenance.

**M2 — `restart_server` uses `os.execv`** (updater.py:403) with `_serve_restart_cmd` stored as a plain list. The command is constructed from CLI args at startup (cli.py:56–59). If the application is somehow invoked with user-controlled arguments that reach `_serve_restart_cmd`, `os.execv` would execute them. In the current code, the list is hardcoded from `sys.executable` + known flags, so this is not exploitable. Noted for future refactoring.

**M3 — Gallery fetch trusts external JSON unconditionally** (web/gallery.py). The repositories listed in `repositories.json` are fetched and their JSON content is parsed and returned to the frontend without any schema validation. A compromised repository could return arbitrarily large payloads or deeply nested structures to cause memory exhaustion or JSON parser issues. Consider adding a size limit on gallery response bodies.

**M4 — `ModUpdate.uid` field in load order API** (api.py:309) accepts an arbitrary string with no validation. The `uid` is written to MO2 mod files. If MO2 parses these files in a way that treats special characters (e.g., `=` or `[section]`) as INI directives, this could corrupt the config. Low risk but worth adding a length limit and character allowlist.

## Test Suite: 403 vs 400/422 Analysis

### Root Cause (Confirmed)

The session token middleware (`auth_middleware` in web/__init__.py:31–38) intercepts **all** `POST`/`PUT`/`DELETE` requests to `/api/*` before FastAPI's request body parsing and Pydantic validation occur. The middleware runs first in the ASGI stack and short-circuits with `403 Invalid session token` when the `X-Session-Token` header is absent.

FastAPI's `TestClient` (backed by `httpx` via Starlette) sends no custom headers by default. The 7 failing tests were all `POST` validation tests that expected `400` or `422` from Pydantic/FastAPI, but received `403` from the middleware instead:

| Test | Expected | Got |
|------|----------|-----|
| `test_open_modlist_invalid_path` | 400 or 404 | 403 |
| `test_open_modlist_traversal_path` | 400 | 403 |
| `test_open_modlist_wrong_extension` | 400 | 403 |
| `test_install_start_workers_too_high` | 422 | 403 |
| `test_install_start_workers_zero` | 422 | 403 |
| `test_install_start_null_byte_in_path` | 422 | 403 |
| `test_install_start_path_traversal_rejected` | 422 | 403 |

### Fix Applied (commit d526ae3)

The correct fix was already applied in the most recent commit: the `client` pytest fixture now passes the session token as a default header:

```python
@pytest.fixture
def client(app):
    """Create a TestClient with session token for authenticated requests."""
    return TestClient(app, headers={"X-Session-Token": SESSION_TOKEN})
```

This is the right approach. The session token is a per-process secret generated at import time (`SESSION_TOKEN = secrets.token_urlsafe(32)`). Tests import it from `wabbajack.web` and inject it into the test client headers — exactly as a real browser would do after calling `GET /api/session`. The security check remains in place; the tests are simply now properly authenticated.

**Do not remove the auth middleware to fix the tests.** The middleware is a valuable defense against DNS rebinding and CSRF on mutating endpoints.

### Recommendation for Future Tests

Any test that calls a `POST`/`PUT`/`DELETE` endpoint must use the session-token-injected `client` fixture (or explicitly pass `headers={"X-Session-Token": SESSION_TOKEN}`). Tests that deliberately test the *unauthenticated* path should use a separate `unauth_client` fixture:

```python
@pytest.fixture
def unauth_client(app):
    """TestClient without session token — for testing 403 responses."""
    return TestClient(app)

def test_post_without_token_returns_403(unauth_client):
    resp = unauth_client.post("/api/install/start", json={...})
    assert resp.status_code == 403
```
