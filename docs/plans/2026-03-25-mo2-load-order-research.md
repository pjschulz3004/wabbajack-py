## 1. Plugin Load Order (plugins.txt & loadorder.txt)

### plugins.txt Format

Location: `<MO2>/profiles/<profile>/plugins.txt` (also `%LocalAppData%/Skyrim Special Edition/plugins.txt` for the engine)

```
# This file is used by Skyrim Special Edition to keep track of your downloaded content.
# Please do not modify this file.
*Skyrim.esm
*Update.esm
*Dawnguard.esm
*HearthFires.esm
*Dragonborn.esm
*Unofficial Skyrim Special Edition Patch.esp
*SkyUI_SE.esp
*EnhancedLightsandFX.esp
SomeDisabledMod.esp
*AnotherMod.esp
```

**Rules:**
- Lines starting with `#` are comments (header)
- `*PluginName.ext` = enabled (active, will load)
- `PluginName.ext` (no asterisk) = disabled (present but won't load)
- One plugin per line
- File is UTF-8 encoded
- Order in the file = load order (top loads first)
- Lists both active AND inactive plugins

### loadorder.txt Format

Location: `<MO2>/profiles/<profile>/loadorder.txt`

```
# This file was automatically generated by Mod Organizer.
Skyrim.esm
Update.esm
Dawnguard.esm
HearthFires.esm
Dragonborn.esm
Unofficial Skyrim Special Edition Patch.esp
SkyUI_SE.esp
EnhancedLightsandFX.esp
AnotherMod.esp
```

**Rules:**
- Header comment from MO2
- Lists ONLY active plugins (no asterisk prefix needed -- all listed are active)
- Includes base game ESMs (which plugins.txt may omit for SSE)
- Order = actual load order the engine will use
- UTF-8 encoded

### How ESM / ESP / ESL Are Handled

**Plugin types determined by TWO factors:** file extension AND header flags in the TES4 record.

| Type | Extension | TES4 Flag | Load Order Slot | Limit |
|------|-----------|-----------|-----------------|-------|
| ESM (Master) | .esm | 0x01 (master) | 0x00-0xFD | 254 total with ESP |
| ESP (Plugin) | .esp | none | 0x00-0xFD | 254 total with ESM |
| ESL (Light) | .esl or .esp with flag | 0x200 (light) | All share 0xFE | 4096 |
| ESP-FE | .esp | 0x01 + 0x200 | Shares 0xFE | Part of 4096 |

**Critical rules:**
- ESMs ALWAYS load before ESPs regardless of plugins.txt order
- ESLs all share load order index 0xFE and don't consume a full slot
- Index 0xFF is reserved for the save game
- Max 254 full plugins (ESM+ESP combined), 4096 light plugins
- ESL plugins limited to 4096 FormID records each
- The engine sorts: hardcoded first, then ESMs, then ESPs -- within each group, the plugins.txt order applies

### TES4 Record Header (Binary Format)

The first record in every plugin file. Critical for determining plugin type.

```
Offset  Size  Field
0x00    4     Signature ("TES4")
0x04    4     Data size (uint32)
0x08    4     Flags (uint32) -- THIS IS KEY
0x0C    4     FormID (always 0 for TES4)
0x10    4     Version control info
0x14    4     Form version + unknown
```

**Flag bits that matter:**
- `0x00000001` = Master file (ESM behavior regardless of extension)
- `0x00000080` = Localized strings
- `0x00000200` = Light master (ESL behavior)

**Python pseudocode to read plugin type:**
```python
import struct

def read_plugin_type(filepath):
    with open(filepath, 'rb') as f:
        sig = f.read(4)  # b'TES4'
        data_size = struct.unpack('<I', f.read(4))[0]
        flags = struct.unpack('<I', f.read(4))[0]

    is_master = bool(flags & 0x01)
    is_light = bool(flags & 0x200)
    is_localized = bool(flags & 0x80)

    ext = filepath.suffix.lower()
    if is_light or ext == '.esl':
        return 'ESL'
    elif is_master or ext == '.esm':
        return 'ESM'
    else:
        return 'ESP'
```

**Master file dependencies** are in MAST subrecords within the TES4 record:
```python
def read_masters(filepath):
    """Read master file dependencies from TES4 header."""
    masters = []
    with open(filepath, 'rb') as f:
        sig = f.read(4)  # TES4
        data_size = struct.unpack('<I', f.read(4))[0]
        f.read(16)  # skip rest of record header

        end_pos = 24 + data_size  # 24 = record header size
        while f.tell() < end_pos:
            sub_type = f.read(4)
            sub_size = struct.unpack('<H', f.read(2))[0]
            if sub_type == b'MAST':
                master_name = f.read(sub_size).rstrip(b'\x00').decode('utf-8')
                masters.append(master_name)
            else:
                f.seek(sub_size, 1)  # skip subrecord data
    return masters
```

## 2. Mod Priority (modlist.txt)

### modlist.txt Format

Location: `<MO2>/profiles/<profile>/modlist.txt`

```
# This file was automatically generated by Mod Organizer.
-Sovngarde - A Nordic Font (Light)
+SMIM - Static Mesh Improvement Mod
+SkyUI
+Unofficial Skyrim Special Edition Patch
+Address Library for SKSE Plugins
-Some Disabled Mod
+Immersive Armors
+DLC: Dawnguard
+DLC: Dragonborn
+DLC: HearthFires
*Unmanaged: Skyrim
```

**Rules:**
- `+Mod Name` = enabled (active in this profile)
- `-Mod Name` = disabled (installed but inactive)
- `*Mod Name` = unmanaged (detected base game / DLC content)
- Lines starting with `#` are comments
- **Bottom of file = HIGHEST priority** (wins conflicts)
- The "Mod Name" matches the directory name in `<MO2>/mods/<Mod Name>/`
- UTF-8 encoded

### Left Pane (Mod Priority) vs Right Pane (Plugin Order)

| Aspect | Left Pane (modlist.txt) | Right Pane (plugins.txt) |
|--------|------------------------|--------------------------|
| Controls | Loose file overwrite priority | Plugin record load order |
| Affects | Textures, meshes, scripts, configs | ESP/ESM record conflicts |
| File | modlist.txt | plugins.txt / loadorder.txt |
| Rule | Higher priority mod's files win | Later-loaded plugin records win |
| Scope | All files in a mod folder | Only .esp/.esm/.esl files |

**Critical design insight:** These are TWO SEPARATE ordering systems that should roughly mirror each other but serve different purposes:

- **Left pane** decides: "When two mods both have `textures/sky/clouds01.dds`, which file does the game see?"
- **Right pane** decides: "When two plugins both modify the same NPC record, which plugin's changes apply?"

Best practice: keep left and right pane orders roughly aligned to avoid situations where Mod A's textures win but Mod B's plugin records win.

### Priority Numbering

MO2 assigns each mod a priority number. Priority 0 is at the top of the list (lowest priority, loaded first). The mod at the bottom has the highest priority number and wins all file conflicts.

In modlist.txt, the first line after comments is priority 0, incrementing downward. But the **bottom entry wins conflicts**, meaning the file is read top-to-bottom with each successive mod overwriting previous ones.

## 3. LOOT Integration

### How LOOT Sorts Plugins

LOOT uses a directed acyclic graph (DAG) + topological sort.

**Graph construction steps:**
1. Separate plugins into 3 graphs: master plugins, non-blueprint masters, non-masters
2. Add edges for each relationship type (in priority order):

| Edge Type | Priority | Description |
|-----------|----------|-------------|
| Hardcoded positions | Hard | Some games hardcode plugin positions (e.g., Skyrim.esm always first) |
| Master dependencies | Hard | Plugin must load after its MAST entries |
| Requirements metadata | Hard | From masterlist/userlist `req` field |
| Load-after metadata | Hard | From masterlist/userlist `load_after` field |
| Group edges | Soft | Plugins in group "Fixes" load before group "Late Loaders" |
| Overlap edges | Soft | Plugins modifying same records/BSA files |
| Tie-break edges | Soft | Lexicographic or existing-order when no other rule applies |

3. Check for cycles (error if found)
4. Topological sort each graph
5. Combine: masters first, then non-masters

**Soft rules** (group, overlap, tie-break) are discarded if they would create a cycle.

### LOOT Masterlist YAML Format

Repository: `github.com/loot/skyrimse` (branch per version like `v0.21`)

```yaml
# Top-level structure

  - 'Invent.Add'
  - 'Keywords'

groups:
  - name: &mainGroup Main Plugins
  - name: &ccGroup Creation Club

  - name: &fixesGroup Fixes & Resources
    after: [*ccGroup]
  - name: &coreGroup Core Mods
    after: [*fixesGroup]
  - name: &lateChangesGroup Late Changes
    after: [*coreGroup]

plugins:
  - name: 'Skyrim.esm'
    group: *mainGroup
    tag:
      - C.Location
      - C.Regions
      - Delev
    dirty:
      - <<: *quickClean
        crc: 0x17AB5E20
        itm: 334
        udr: 92

  - name: 'Unofficial Skyrim Special Edition Patch.esp'
    group: *fixesGroup
    tag:
      - Actors.ACBS
      - Delev
      - Relev
    msg:
      - type: say
        content: 'Requires all DLCs.'

  - name: 'SomePlugin\.esp'  # regex supported
    req:
      - name: 'RequiredPlugin.esm'
        display: '[Required Mod](https://nexusmods.com/...)'
    inc:
      - name: 'IncompatibleMod.esp'
    load_after:
      - name: 'ShouldLoadFirst.esp'
        condition: 'file("OptionalDep.esp")'
    tag:
      - name: -Relations  # minus = remove tag
        condition: 'file("SomeOtherMod.esm")'
```

**Key metadata fields per plugin:**
- `name` - Plugin filename (supports regex)
- `group` - Load order group assignment
- `req` - Required plugins (hard dependency)
- `inc` - Incompatible plugins (error if both active)
- `load_after` - Soft ordering constraint
- `msg` - User-facing messages (type: say/warn/error)
- `tag` - Bash Tag suggestions (for Wrye Bash patching)
- `dirty` - ITM/UDR counts with CRC for specific dirty versions
- `clean` - CRC values of known-clean versions
- `condition` - Conditional logic: `file("X.esp")`, `version("X.esp", "1.0", >=)`, `active("X.esp")`

## 4. Virtual Filesystem (USVFS) & Linux Alternatives

### How USVFS Works (Windows)

USVFS = User Space Virtual File System. Windows-only, GPL3.

**Mechanism:**
1. MO2 builds a mapping: virtual path -> real path (from mod priority order)
2. Injects `usvfs_xx.dll` into the game process via DLL injection
3. DLL hooks low-level NT API calls (`NtCreateFile`, `NtOpenFile`, `NtQueryDirectoryFile`)
4. When the game reads `Data/textures/sky.dds`, USVFS intercepts and redirects to the winning mod's actual file location
5. Process-local: only the game process sees the virtual filesystem

**Result:** No files are ever copied. Each mod stays in its own directory. The game "sees" a merged Data folder that doesn't physically exist.

### Linux Alternatives

**Option A: OverlayFS (kernel-level, recommended)**

OverlayFS is in the mainline Linux kernel (since 3.18). It creates a union of multiple directories.

```bash
# Structure:
#   lower layers = mod directories (read-only, in priority order)
#   upper layer  = writable layer for game saves/generated files
#   work dir     = OverlayFS internal bookkeeping
#   merged       = what the game sees

mount -t overlay overlay \
  -o lowerdir=/mods/highest_priority:/mods/next:/mods/base_game,\
     upperdir=/game/writeable,\
     workdir=/game/.work \
  /game/merged

# Or with modern fsconfig API (bypasses 4096-byte mount option limit):
# Use fsconfig(fd, FSCONFIG_SET_STRING, "lowerdir+", path, 0)
# for each mod layer
```

**Advantages for modding:**
- Zero disk duplication (like USVFS)
- Kernel-level performance (faster than USVFS hooking)
- Trivial mod-pack switching (just remount with different layers)
- Non-destructive: base game files untouched
- Lower layers are read-only (mods can't accidentally corrupt)

**Challenges:**
- Requires root for mounting (can use `unshare --map-root-user --mount` for unprivileged)
- `lowerdir` option has a 4096-byte limit in traditional `mount()` -- problematic with 1000+ mods
- Modern `fsconfig()` API bypasses the limit using `lowerdir+` parameter
- Layer count has practical limits (kernel stack depth)

**Option B: Hardlinks (current approach in deploy-bg3-mods.sh)**

```bash
# For each mod file, create a hardlink in the game's Data directory
ln -f /mods/SomeTextureMod/textures/sky.dds /game/Data/textures/sky.dds
```

- Works with Proton/Wine (unlike symlinks which Proton can't follow)
- Uses disk space for inode entries but no file duplication
- Must re-link when mod order changes
- Doesn't support same-file-different-content (can't have two versions)

**Option C: FUSE (Filesystem in Userspace)**

Write a custom FUSE filesystem that presents a merged view. More complex but fully userspace (no root needed). Python FUSE bindings exist (`fusepy`).

**Recommendation for wabbajack-py:** Use OverlayFS as primary strategy (most robust, kernel-supported), with hardlink fallback for systems where OverlayFS isn't available or when running under Proton where the game needs real filesystem entries.

## 5. Profile System

### Per-Profile Directory Structure

```
<MO2>/profiles/<ProfileName>/
  modlist.txt          # Mod enable/disable state + priority order
  plugins.txt          # Plugin enable/disable state
  loadorder.txt        # Active plugin load order
  lockedorder.txt      # Plugins with manually locked positions (UTF-8)
  archives.txt         # BSA load order (one filename per line, UTF-8)
  initweaks.ini        # Merged INI tweaks (auto-generated, local 8-bit)
  Skyrim.ini           # Profile-specific game INI (if using profile INIs)
  SkyrimPrefs.ini      # Profile-specific prefs INI (if using profile INIs)
```

### What's Per-Profile vs Shared

| Per-Profile | Shared Across Profiles |
|-------------|----------------------|
| modlist.txt (which mods active + order) | Actual mod files in `<MO2>/mods/` |
| plugins.txt (which plugins active) | Downloaded archives in `<MO2>/downloads/` |
| loadorder.txt (plugin order) | Mod metadata (meta.ini in each mod folder) |
| INI files (if profile-specific INIs enabled) | LOOT masterlist/userlist |
| Save games (if profile-specific saves enabled) | Tool configurations |
| lockedorder.txt | |
| archives.txt | |
| initweaks.ini | |

### archives.txt Format

```
# BSAs to register, one per line, sorted by priority
# Later entries overwrite earlier ones (same as modlist.txt convention)
Skyrim - Textures0.bsa
Skyrim - Textures1.bsa
Unofficial Skyrim Special Edition Patch - Textures.bsa
SMIM-SE-Meshes.bsa
```

### Profile Creation Behavior

When creating a new profile, MO2:
1. Creates the directory under `profiles/`
2. Copies INI files from the global location (Documents/My Games/Skyrim SE/)
3. INI copies are snapshots -- later global INI changes don't propagate to existing profiles
4. modlist.txt is initialized with all mods disabled or copied from a template profile

## 6. Conflict Resolution

### How MO2 Detects and Shows Conflicts

**File-level conflict detection:**
For every file in every enabled mod, MO2 builds a virtual directory tree. When two mods contain the same relative path (e.g., `textures/architecture/farmhouse01.dds`), that's a conflict.

**Resolution rule:** The mod with HIGHER priority (lower in modlist.txt) wins. Its file is what the virtual filesystem presents.

**Visual indicators in MO2:**
- Green `+` icon (lightning bolt) = mod is **overwriting** files from other mods (this mod wins)
- Red `-` icon (lightning bolt) = mod has files being **overwritten** by higher-priority mods (this mod loses)
- Both can appear simultaneously (mod wins some conflicts, loses others)

**Conflicts tab** for each mod shows:
- "Winning" files: files this mod provides that override lower-priority mods
- "Losing" files: files in this mod that are overridden by higher-priority mods
- For each conflict: the competing mod name and the file path

### Implementation Approach for wabbajack-py

```python
from collections import defaultdict
from dataclasses import dataclass, field

@dataclass
class FileConflict:
    relative_path: str
    winner: str          # mod name that wins
    losers: list[str]    # mod names that lose

def detect_conflicts(modlist: list[tuple[str, str]], mods_dir: Path) -> dict:
    """
    modlist: [('+', 'ModName'), ('-', 'DisabledMod'), ...] from modlist.txt
    Bottom of list = highest priority

    Returns dict of relative_path -> FileConflict
    """
    # file_path -> list of (priority, mod_name)
    file_owners: dict[str, list[tuple[int, str]]] = defaultdict(list)

    for priority, (state, mod_name) in enumerate(modlist):
        if state != '+':
            continue
        mod_path = mods_dir / mod_name
        for file_path in mod_path.rglob('*'):
            if file_path.is_file():
                rel = file_path.relative_to(mod_path)
                file_owners[str(rel)].append((priority, mod_name))

    conflicts = {}
    for rel_path, owners in file_owners.items():
        if len(owners) > 1:
            owners.sort(key=lambda x: x[0])  # sort by priority
            winner = owners[-1][1]  # highest priority wins
            losers = [name for _, name in owners[:-1]]
            conflicts[rel_path] = FileConflict(rel_path, winner, losers)

    return conflicts
```

### Recommended Approach: Roll Your Own

For wabbajack-py, the header parsing is simple enough to implement directly (see Section 1 pseudocode). The critical operations are:

1. **Read TES4 header** (24 bytes + subrecords) -- ~50 lines of Python
2. **Extract flags** (master, light, localized) -- bitmask operations
3. **Extract MAST dependencies** -- iterate subrecords
4. **Parse plugins.txt / modlist.txt** -- simple text parsing
5. **Build dependency graph** -- standard DAG from master dependencies
6. **Topological sort** -- Python stdlib or ~20 lines

**Don't need full plugin parsing.** Only the TES4 header record matters for load order. No need to parse the thousands of other records (NPCs, items, quests, etc.).

### Minimal Plugin Header Parser

```python
import struct
from pathlib import Path
from dataclasses import dataclass

@dataclass
class PluginHeader:
    filepath: Path
    filename: str
    flags: int
    is_master: bool
    is_light: bool
    is_localized: bool
    masters: list[str]
    description: str
    author: str

    @property
    def plugin_type(self) -> str:
        if self.is_light or self.filepath.suffix.lower() == '.esl':
            return 'ESL'
        elif self.is_master or self.filepath.suffix.lower() == '.esm':
            return 'ESM'
        return 'ESP'

def parse_plugin_header(filepath: Path) -> PluginHeader:
    """Parse only the TES4 header record from a Bethesda plugin file."""
    masters = []
    description = ''
    author = ''

    with open(filepath, 'rb') as f:
        # Record header (24 bytes for Skyrim)
        sig = f.read(4)
        if sig != b'TES4':
            raise ValueError(f"Not a valid plugin: {filepath}")

        data_size = struct.unpack('<I', f.read(4))[0]
        flags = struct.unpack('<I', f.read(4))[0]
        f.read(12)  # formID + version control + form version

        # Parse subrecords within TES4
        end_pos = 24 + data_size
        while f.tell() < end_pos:
            sub_type = f.read(4)
            sub_size = struct.unpack('<H', f.read(2))[0]

            if sub_type == b'MAST':
                master = f.read(sub_size).rstrip(b'\x00').decode('utf-8')
                masters.append(master)
            elif sub_type == b'SNAM':
                description = f.read(sub_size).rstrip(b'\x00').decode('utf-8', errors='replace')
            elif sub_type == b'CNAM':
                author = f.read(sub_size).rstrip(b'\x00').decode('utf-8', errors='replace')
            else:
                f.seek(sub_size, 1)

    return PluginHeader(
        filepath=filepath,
        filename=filepath.name,
        flags=flags,
        is_master=bool(flags & 0x01),
        is_light=bool(flags & 0x200),
        is_localized=bool(flags & 0x80),
        masters=masters,
        description=description,
        author=author,
    )
```

### Load Order Sorting (Simplified LOOT-like)

```python
from collections import defaultdict

def sort_plugins(plugin_headers: list[PluginHeader],
                 current_order: list[str]) -> list[str]:
    """
    Sort plugins respecting:
    1. Masters before non-masters
    2. Master dependencies (MAST entries)
    3. Preserve relative order from current_order where possible
    """
    # Build dependency graph
    by_name = {p.filename.lower(): p for p in plugin_headers}
    graph = defaultdict(set)  # plugin -> set of plugins it must load after

    for plugin in plugin_headers:
        for master in plugin.masters:
            if master.lower() in by_name:
                graph[plugin.filename.lower()].add(master.lower())

    # Separate into masters and non-masters
    masters = [p for p in plugin_headers if p.plugin_type in ('ESM', 'ESL')]
    non_masters = [p for p in plugin_headers if p.plugin_type == 'ESP']

    def topo_sort(plugins):
        """Topological sort preserving relative order where possible."""
        names = {p.filename.lower() for p in plugins}
        # Filter graph to only include plugins in this subset
        local_deps = {
            p.filename.lower(): graph[p.filename.lower()] & names
            for p in plugins
        }

        sorted_list = []
        visited = set()
        temp_mark = set()

        # Use current_order for tie-breaking
        order_index = {
            name.lower(): i for i, name in enumerate(current_order)
        }
        plugins_sorted = sorted(
            plugins,
            key=lambda p: order_index.get(p.filename.lower(), 999999)
        )

        def visit(name):
            if name in temp_mark:
                raise ValueError(f"Cyclic dependency involving {name}")
            if name in visited:
                return
            temp_mark.add(name)
            for dep in local_deps.get(name, set()):
                visit(dep)
            temp_mark.discard(name)
            visited.add(name)
            sorted_list.append(name)

        for p in plugins_sorted:
            visit(p.filename.lower())

        return sorted_list

    sorted_masters = topo_sort(masters)
    sorted_non_masters = topo_sort(non_masters)

    # Reconstruct with original casing
    name_map = {p.filename.lower(): p.filename for p in plugin_headers}
    return [name_map[n] for n in sorted_masters + sorted_non_masters]
```
