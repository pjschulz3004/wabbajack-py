## Best Practices Found

### FastAPI

#### Middleware Ordering (Critical)
CORS middleware **must be first** in the stack. The current code adds CORSMiddleware first, then adds a custom `auth_middleware` via `@app.middleware("http")`. This ordering is correct — custom `@app.middleware` handlers wrap *around* route handlers but are inserted *after* `add_middleware()` calls in Starlette's stack, meaning CORS runs first on the way in. No action needed on ordering.

However: **the session token approach (Origin/Referer gating) is insufficient alone against DNS rebinding**.

#### DNS Rebinding — Current Approach is Incomplete
The current `/api/session` endpoint checks `Origin` and `Referer` headers to gate the session token. This is partially correct but has a gap:

- DNS rebinding attacks bypass CORS because the browser treats the attacker's domain as "same origin" after the rebind. At that point, there is no `Origin` header to check — the request looks same-origin.
- The `Origin`/`Referer` check **does** protect against simple CSRF from a remote page loading before the rebind.
- For a local-only tool, the correct defense is **`TrustedHostMiddleware`** which validates the `Host` header.

**Recommended addition** (add *before* CORSMiddleware in the stack):
```python
from starlette.middleware.trustedhost import TrustedHostMiddleware
app.add_middleware(TrustedHostMiddleware, allowed_hosts=["localhost", "127.0.0.1"])
```

This stops DNS rebinding because an attacker domain resolves to 127.0.0.1 but the `Host` header still reads `attacker.com` — TrustedHostMiddleware rejects it with 400.

Note: TrustedHostMiddleware matches the hostname only, not port. `localhost:6969` passes if `"localhost"` is in the list.

CVE-2026-32632 was issued against Glances for exactly this omission in a FastAPI/uvicorn local app.

#### Lifespan API (Replace deprecated on_event)
`@app.on_event("startup")` / `@app.on_event("shutdown")` are deprecated since FastAPI 0.95.0. The project doesn't use them currently, but any future startup logic (e.g., initializing a shared httpx.AsyncClient) should use the lifespan pattern:

```python
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    # startup
    app.state.http_client = httpx.AsyncClient()
    yield
    # shutdown
    await app.state.http_client.aclose()

app = FastAPI(lifespan=lifespan)
```

#### WebSocket Heartbeat (Missing)
The current WebSocket implementation in `ws.py` has no ping/pong heartbeat. If a client disconnects without sending a close frame (browser crash, network loss), the server keeps trying to broadcast to a dead connection. Current code does catch errors during broadcast and prunes dead clients — this is correct — but silent dead connections can persist between broadcasts indefinitely.

**Best practice:** Add a server-sent ping every 30 seconds. Drop clients that don't respond within N seconds. The existing broadcast cleanup handles the dead-client pruning, but a heartbeat catches connections that never trigger a broadcast.

```python
# In websocket_endpoint, add a background heartbeat:
async def heartbeat():
    while True:
        await asyncio.sleep(30)
        for ws in list(_clients):
            try:
                await ws.send_text(json.dumps({"type": "ping"}))
            except Exception:
                _clients.discard(ws)
```

#### CORS allow_credentials
The current CORS config does not set `allow_credentials=True`. This is correct — with `allow_credentials=True` you cannot use `allow_origins=["*"]`, but the current setup uses explicit origin list so adding credentials would be fine if needed. No action needed unless cookie-based auth is added later.

#### `requests` vs `httpx` Inconsistency
`auth.py` uses `import requests` (sync) for token validation while the rest of the app uses httpx. When running inside an async FastAPI endpoint (like `set_nexus_token`), the blocking `requests.get()` call blocks the event loop. Should be replaced with an async httpx call.

### Nexus Mods API

#### SSO Broken: Root Cause Identified
The "application ID was invalid" error is a **registration problem**, not a code bug.

The Nexus Mods SSO WebSocket flow requires an **approved `appid`/application slug** that is assigned by Nexus Mods staff. There is no self-service registration portal.

The current code sends:
```python
auth_url = f"https://www.nexusmods.com/sso?id={request_id}&application=wabbajack-py"
```

The `application=wabbajack-py` slug is **unregistered**. Nexus will reject this with "application ID was invalid". The SSO system only works for apps that have been approved by Nexus Mods Community Managers.

**The correct SSO message format** also includes `appid` in the WebSocket handshake:
```json
{ "id": "uuid-here", "token": null, "protocol": 2, "appid": "your_registered_slug" }
```

The current implementation sends `token` but not `appid` in the WS message — this is also wrong.

**Two-phase SSO response**: The server first sends a `connection_token` (for reconnects), then after user auth sends the `api_key`. The current code only waits for a single WS message and expects `api_key` immediately — this will fail.

#### Three Authentication Options (in order of feasibility)

**Option 1 — Manual API Key entry (works NOW, no registration needed)**
The user pastes their personal API key from `https://www.nexusmods.com/users/myaccount?tab=api+access`. This already exists in the codebase. This is the correct fallback for unregistered tools.

**Option 2 — SSO WebSocket (requires Nexus staff approval)**
Contact [email protected] or Nexus Mods Discord `#api-discussion` channel. Request an `appid` for the tool. This is free but requires approval. With a registered app slug, the SSO flow works.

**Option 3 — OAuth 2.0 with PKCE (also requires registration, is the future)**
Nexus is migrating from API keys to OAuth 2.0. No self-service registration UI exists yet. Process: email [email protected] with app name, description, and redirect URIs.
- Authorization URL: `https://users.nexusmods.com/oauth/authorize`
- Token URL: `https://users.nexusmods.com/oauth/token`
- OpenID config: `https://users.nexusmods.com/.well-known/openid-configuration`
- For desktop apps: use `localhost:RANDOMPORT/callback` as redirect URI
- API keys are "slowly being deprecated" in favor of OAuth 2.0

#### API v2 (GraphQL) — Migration Incoming
Nexus has turned off their legacy API and replaced it with a new v2 API + GraphQL endpoint at `https://graphql.nexusmods.com/`. The v1 REST API (`api.nexusmods.com/v1/`) is being phased out.

- Most GraphQL v2 queries work without auth
- Download link endpoints still require auth (API key or OAuth token)
- The current v1 endpoint (`/v1/games/{game}/mods/{mod_id}/files/{file_id}/download_link.json`) may stop working

**Rate limits (v1):**
- 20,000 requests per 24-hour period (combined all endpoints)
- After limit: 500 requests per hour
- Headers: `X-RL-Daily-Remaining`, `X-RL-Hourly-Remaining`
- The current code reads `X-RL-Daily-Remaining` correctly

**Current code issues in `nexus.py`:**
- Uses `urllib.request` not httpx — should migrate
- The `NEXUS_RATE_DELAY = 0.3` (300ms per API call) is sensible but aggressive; increase to 0.5s to be safer
- API calls are serialized via `api_lock` while downloads are parallel — this is the correct pattern

### httpx

#### Current Usage Gap
The project lists `httpx>=0.25` as a dependency but the actual downloaders (`nexus.py`, `cdn.py`) use `urllib.request`. Only the web module likely uses httpx. This is an inconsistency — either commit to httpx or remove it from dependencies.

#### Connection Pooling Best Practices
- Create one `AsyncClient` instance for the app lifetime (in lifespan), not per-request
- For sync downloads in ThreadPoolExecutor: use one `httpx.Client()` per thread or a shared client with appropriate pool size
- `httpx.Limits(max_connections=20, max_keepalive_connections=10)` for download pools

#### Streaming Large Files (the right pattern)
```python
async with client.stream("GET", url) as response:
    response.raise_for_status()
    total = int(response.headers.get("content-length", 0))
    downloaded = 0
    async with aiofiles.open(dest, "wb") as f:
        async for chunk in response.aiter_bytes(chunk_size=65536):
            await f.write(chunk)
            downloaded += len(chunk)
            # Use response.num_bytes_downloaded for actual progress
            # (accounts for Content-Encoding/compression)
```

Key: use `response.num_bytes_downloaded` (not `len(chunk)`) for accurate progress with compressed responses.

#### Timeout Configuration for Downloads
```python
# For large downloads: no read timeout (file could be slow), but connect timeout
timeout = httpx.Timeout(connect=10.0, read=None, write=None, pool=5.0)
```

The default 5-second timeout will kill large downloads. Set `read=None` for downloads.

#### Retry Pattern
httpx does not retry by default. Use `tenacity` or a manual loop:
```python
for attempt in range(MAX_RETRIES):
    try:
        async with client.stream("GET", url, timeout=timeout, follow_redirects=True) as r:
            ...
            break
    except (httpx.TransportError, httpx.TimeoutException) as e:
        if attempt == MAX_RETRIES - 1: raise
        await asyncio.sleep(2 ** attempt)
```

### Svelte 5 / Frontend

#### Stores vs Runes
`frontend/src/lib/stores/ws.ts` uses `writable()` stores from `svelte/store`. **Stores are NOT deprecated in Svelte 5** — they will continue to work. However, the recommended Svelte 5 pattern for shared state is to use `$state` in `.svelte.ts` files.

The current stores pattern is functionally fine. The main benefit of migrating to runes would be better TypeScript inference and the ability to use the same reactive primitives inside and outside components. Not urgent.

**If migrating:**
```typescript
// ws.svelte.ts (not ws.ts)
let _connected = $state(false);
let _logs = $state<WsMessage[]>([]);
export const connected = { get value() { return _connected; } };
```

#### $effect Anti-patterns
The current App.svelte uses `$effect(() => { connectWs(); })` which is correct but runs on every reactive dependency change. Since `connectWs()` guards against re-connection at the top, this is harmless but slightly noisy. A `$effect.root()` or mounting logic in `onMount` equivalent would be cleaner.

**Known Svelte 5 issue:** Nested object state with `$effect` can trigger `ERR_SVELTE_TOO_MANY_UPDATES`. If this occurs, restructure state as classes or use `$derived.by()` with explicit dependency tracking.

#### vite-plugin-svelte v7 Changes
- v7 requires Svelte 5.46.4+. The project is at `^5.53.12` — compatible.
- v7 removes `cssHash` override (Svelte 5 now generates stable CSS hashes internally)
- CSS hash behavior change is transparent; no code changes needed

#### Missing: WebSocket heartbeat on client side
`ws.ts` reconnects every 3 seconds on close but has no mechanism to detect a stale-open connection (server died without closing WS). Adding a client-side ping check would improve reliability:
```typescript
// Check for server ping messages, reset timer; if no ping in 45s, reconnect
```

## Security Notes

### Session Token Design (Current)
The `X-Session-Token` mechanism protects mutating endpoints against CSRF. The token is issued via `/api/session` which checks Origin/Referer. This is reasonable but:
- Has no expiry (token lives for the server process lifetime) — fine for a local tool
- Should be complemented by TrustedHostMiddleware (see above)
- Does NOT protect against a locally running malicious process that can call `localhost:6969` directly — but that's an acceptable threat model for a gaming tool

### WebSocket Security
The WS endpoint at `/ws` has no auth. Any local process can connect. This is acceptable for a local-only tool but means the WS should not expose sensitive data (API keys etc.) in messages. Currently it does not.
